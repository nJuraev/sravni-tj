# PRD: Агрегатор банковских продуктов Таджикистана — MVP

**Версия:** 1.0
**Дата:** 2026-06-04
**На основе:** ТЗ.md

---

## Problem Statement

Пользователи в Таджикистане не имеют единой площадки для сравнения банковских продуктов (кредитов и депозитов) разных банков. Чтобы выбрать лучший вклад или кредит, нужно вручную обходить сайты до 20 банков, сравнивать разрозненные данные и самостоятельно считать итоговый доход/платёж. Процесс оформления заявки также требует звонить или приходить в банк.

---

## Solution

Веб-платформа **Sravni.tj** — агрегатор, который автоматически собирает актуальные условия по кредитам и депозитам с сайтов банков Таджикистана, отображает их в едином каталоге с фильтрами и сортировкой, предоставляет встроенный калькулятор и позволяет оставить заявку прямо на сайте. Данные обновляются по расписанию без участия человека.

---

## User Stories

### Просмотр и сравнение продуктов

1. Как посетитель сайта, я хочу видеть каталог кредитов всех банков на одной странице, чтобы не заходить на сайт каждого банка по отдельности.
2. Как посетитель сайта, я хочу видеть каталог депозитов всех банков на одной странице, чтобы выбрать наиболее выгодный вклад.
3. Как посетитель сайта, я хочу видеть карточку продукта с информацией о банке, процентной ставке, сумме, сроке и условиях, чтобы понять основные параметры предложения.
4. Как посетитель сайта, я хочу сравнить несколько продуктов разных банков бок о бок, чтобы наглядно увидеть разницу в условиях.
5. Как посетитель сайта, я хочу фильтровать кредиты и депозиты по сумме, сроку, валюте и процентной ставке, чтобы быстро найти подходящие предложения.
6. Как посетитель сайта, я хочу фильтровать продукты по дополнительным признакам (онлайн-оформление, без поручителя, капитализация, возможность пополнения), чтобы сузить выбор по важным для меня критериям.
7. Как посетитель сайта, я хочу сортировать список продуктов по процентной ставке и другим параметрам, чтобы видеть лучшие предложения сверху.
8. Как посетитель сайта, я хочу видеть ставки в трёх валютах (TJS, USD, EUR), чтобы сравнивать продукты в нужной мне валюте.
9. Как посетитель сайта, я хочу переключить интерфейс между русским и таджикским языком, чтобы пользоваться сайтом на удобном мне языке.
10. Как посетитель сайта, я хочу видеть названия и описания банков и продуктов на выбранном мной языке, чтобы вся информация была локализована.

### Калькулятор

11. Как посетитель сайта, я хочу рассчитать ежемесячный платёж по кредиту (сумма, срок, ставка), чтобы оценить финансовую нагрузку.
12. Как посетитель сайта, я хочу рассчитать доход по вкладу (сумма, срок, ставка, капитализация), чтобы понять, сколько заработаю.
13. Как посетитель сайта, я хочу видеть результаты калькулятора в режиме реального времени при изменении параметров, чтобы не нажимать дополнительную кнопку.

### Заявки (лиды)

14. Как посетитель сайта, я хочу оставить заявку на кредит или депозит прямо на сайте, чтобы не звонить в банк самостоятельно.
15. Как посетитель сайта, я хочу заполнить форму заявки с минимальными данными (ФИО, телефон), чтобы процесс был быстрым.
16. Как посетитель сайта, я хочу дать согласие на обработку персональных данных через чекбокс со ссылкой на политику, чтобы понимать, как будут использованы мои данные.
17. Как посетитель сайта, я не хочу, чтобы моя заявка была отправлена без явного согласия на обработку персональных данных, чтобы мои права были защищены.
18. Как банк, я хочу получать заявки клиентов на свои продукты по электронной почте, чтобы обрабатывать их в своей CRM.

### Парсер / обновление данных

19. Как система, я хочу автоматически забирать актуальные условия кредитов и депозитов с сайтов банков по расписанию, чтобы данные на платформе всегда были актуальны.
20. Как система, я хочу конвертировать HTML-страницу банка в Markdown и передавать в AI-модель для структурированного извлечения данных, чтобы минимизировать хрупкость парсера к изменению вёрстки.
21. Как система, я хочу получать от AI строго типизированный JSON по заданной схеме (ставка, срок, сумма, валюта, фичи), чтобы исключить финансовые галлюцинации.
22. Как система, я хочу записывать распарсенные продукты в PostgreSQL и проставлять им статус, чтобы фронтенд показывал только актуальные данные.
23. Как разработчик, я хочу включить конфиг-флаг логирования метаданных парсинга (URL-источник, время, сырой ответ AI, статус запуска), чтобы отлаживать ошибки извлечения данных при необходимости.
24. Как система, я хочу корректно парсить банк с несколькими URL по категориям (напр. `deposit.dc.tj` для депозитов и `credit.dc.tj` для кредитов), чтобы данные по всем продуктам банка были собраны.
25. Как система, я хочу учитывать зависимость ставки от срока, суммы и валюты при извлечении данных, чтобы не потерять многоуровневые тарифные сетки банков.

### Управление данными

26. Как администратор БД, я хочу менять статус продукта (активен / черновик / скрыт / устарел) напрямую в базе данных, чтобы скрыть некорректные или неактуальные данные без развёртки новой версии.
27. Как администратор БД, я хочу менять статус банка (активен / неактивен), чтобы временно скрыть все продукты банка при необходимости.
28. Как администратор БД, я хочу добавить или отключить URL-источник для парсинга конкретного банка и категории, не делая деплой, чтобы оперативно реагировать на изменения на сайтах банков.
29. Как система, я хочу хранить признак «банк-партнёр» у каждого банка, чтобы в будущем дифференцировать партнёрские интеграции.

## Implementation Decisions

### Монорепо-структура

```
sravni-tj/
├── parser/           # Go — парсер продуктов (по крону)
├── backend/          # PHP / Laravel — REST API + приём заявок
├── frontend/         # Vue.js — витрина сравнения + формы заявок
└── docker-compose.yml
```

### БД (PostgreSQL) — ключевые сущности

**`banks`** — банки:
- `id`, `name_ru`, `name_tg`, `email`, `status` (active/inactive), `is_partner`, `created_at`, `updated_at`

**`bank_source_urls`** — URL для парсинга, привязанные к банку и категории продукта:
- `id`, `bank_id → banks.id`, `category` ENUM('credit', 'deposit'), `url`, `is_active`, `created_at`, `updated_at`
- Один банк может иметь несколько строк — отдельный URL на кредиты и на депозиты.
- Добавить/отключить источник можно через прямой UPDATE в БД без деплоя.

**`products`** — продукты:
- `id`, `bank_id`, `source_url_id → bank_source_urls.id`, `category` (credit/deposit), `name_ru`, `name_tg`, `status` (active/draft/hidden/outdated), `currency` (TJS/USD/EUR), `rate_min`, `rate_max`, `amount_min`, `amount_max`, `term_min`, `term_max`, `features` (jsonb), `parsed_at`, `created_at`, `updated_at`
- Ставки, зависящие от нескольких параметров (срок × сумма × валюта), хранятся либо в связанной таблице `product_rates`, либо в структурированном jsonb-поле — решается при проектировании схемы.

**`leads`** — заявки:
- `id`, `product_id`, `bank_id`, `full_name`, `phone`, `consent` (boolean, обязательно true), `created_at`

**`parser_runs`** — метаданные парсинга (только при включённом флаге):
- `id`, `bank_source_url_id → bank_source_urls.id`, `started_at`, `finished_at`, `ai_raw_response`, `status`, `error_message`

### Парсер (Go)

- При запуске делает `SELECT` из `bank_source_urls WHERE is_active = true` — получает список задач.
- Для каждой задачи: загрузка страницы → конвертация в Markdown (Firecrawl / Jina) → вызов AI-модели (Gemini / Qwen) в режиме structured output → валидация JSON по схеме → запись в `products`.
- AI вызывается в строго типизированном JSON-режиме (structured output / function calling); числовые поля валидируются на диапазон (ставка 0–100%, сумма > 0) до записи в БД.
- Конфиг-флаг `PARSER_DEBUG_LOG=true`: при включении каждый запуск логируется в `parser_runs` с сырым ответом AI. По умолчанию — отключён.
- Многодоменные банки обрабатываются естественно: у банка просто несколько строк в `bank_source_urls` с разными URL и категориями.

### Backend (Laravel)

- REST API:
  - `GET /api/products` — список с фильтрами (category, amount, term, currency, rate_min, rate_max, features), только `status = active` и банк `status = active`.
  - `GET /api/products/{id}` — карточка продукта.
  - `GET /api/banks` — список активных банков.
  - `POST /api/leads` — приём заявки.
- Фильтры реализуются через Laravel Query Builder, без raw SQL.
- `POST /api/leads`: валидация (ФИО, телефон, product_id, `consent` обязателен и должен быть `true`) → сохранение в `leads` → отправка email на адрес банка (`banks.email`) через Laravel Mail.
- Поле `is_partner` хранится, но на маршрутизацию заявок не влияет — все заявки идут на email.

### Frontend (Vue.js)

- i18n: два locale (`ru`, `tg`) через Vue I18n. Мультиязычные поля (`name_ru`/`name_tg`, `description_ru`/`description_tg`) отображаются по активному locale.
- Калькулятор: вычисления полностью на клиенте — аннуитетная формула для кредита, простые/сложные проценты для депозита. Данные берутся из карточки продукта или вводятся вручную.
- Сравнение: до 3–4 продуктов одновременно, состояние в Pinia store.
- Форма заявки: чекбокс consent обязателен; кнопка submit `disabled` без галочки.
- Дизайн: цветовая палитра по референсу eskhata.com (см. раздел ниже).

## Testing Decisions

Хороший тест проверяет внешнее поведение модуля (HTTP-ответ, email-отправку, результат фильтрации) — не детали реализации (приватные методы, внутренние SQL-запросы).

### Parser (Go)

- Юнит-тесты AI-схемы: подаём корректный и некорректный JSON — проверяем валидацию типов и диапазонов.
- Юнит-тесты конфиг-флага: при `PARSER_DEBUG_LOG=true` запись в `parser_runs` создаётся, при `false` — нет.
- Интеграционные тесты против тестовой PostgreSQL: парсер читает `bank_source_urls`, пишет продукты — запись присутствует с корректными полями.

### Backend (Laravel)

Feature-тесты с `RefreshDatabase`:
- `GET /api/products` возвращает только продукты со `status = active` и активным банком.
- Фильтры по категории, валюте, сумме, сроку, ставке корректно сужают выборку.
- `POST /api/leads` без `consent = true` возвращает 422.
- `POST /api/leads` с валидными данными → запись в `leads` + `Mail::fake()` перехватывает письмо на email банка.

### Frontend (Vue.js)

- Юнит-тесты (Vitest + Vue Test Utils):
  - Калькулятор: числовые результаты при разных входных параметрах.
  - Форма заявки: кнопка `disabled` без consent, `enabled` с consent.
  - i18n: переключение locale меняет отображаемый текст.
- E2E (Playwright):
  - Пользователь применяет фильтры → список продуктов обновляется.
  - Пользователь добавляет продукты в сравнение → страница сравнения отображает их бок о бок.
  - Пользователь заполняет форму заявки и отправляет → видит сообщение об успехе.

## Out of Scope

- Авторизация и личный кабинет пользователей.
- Административная панель (управление через прямой доступ к БД на MVP).
- Ручное ревью распарсенных данных.
- Категории продуктов: страхование, ипотека, карты, переводы — только кредиты и депозиты на MVP.
- Мобильное приложение.
- Платёжная интеграция.
- API/webhook-интеграция с банками для доставки заявок — только email на MVP.
- Полная схема БД (проектируется отдельно).

## Further Notes

- **Точный список банков** (до 20) и их URL предоставляет заказчик — без него парсер не может быть запущен в prod. URL заносятся в `bank_source_urls` напрямую в БД.
- **Политика обработки персональных данных** — текст составляется перед запуском; исполнитель готовит черновик. Является юридическим требованием для формы заявки.
- **AI-модель для парсинга** — Gemini/Qwen используются в бесплатном tier; при росте нагрузки или ухудшении качества экстракции может потребоваться переход на платный tier или смена провайдера.
- **Сложные тарифные сетки:** ставка может зависеть от суммы И срока одновременно; AI-схема должна поддерживать массив диапазонов, а не одно числовое поле. Финальное решение по хранению — при проектировании схемы БД.
- **Будущее:** API/webhook-интеграция с банками-партнёрами для доставки заявок (помимо email) — возможна по мере появления партнёров.
