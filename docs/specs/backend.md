# Спецификация: Backend (Laravel)

**Модуль:** `backend/`
**Технология:** PHP / Laravel (10/11+)
**Назначение:** REST API для витрины (чтение продуктов/банков) + приём заявок (leads) с доставкой на email банка.
**Статус:** контракт фундамента. Описывает ВНЕШНЕЕ поведение — эндпоинты, бизнес-правила, валидацию, ошибки. Точный формат запросов/ответов JSON — в `docs/api/contracts.md` (замороженный контракт). Здесь — правила, инварианты и связь со схемой БД.

---

## 1. Назначение и границы

- **Делает:** читает `products`, `banks` из общей PostgreSQL и отдаёт через REST; принимает заявки, пишет в `leads`, отправляет email на `banks.email`.
- **НЕ делает:** не пишет в `products`/`banks`/`bank_source_urls` (это парсер/администратор); не парсит; не рендерит UI; не имеет авторизации и админки (вне MVP).
- **Только чтение для каталога:** backend никогда не модифицирует продукты и банки. Единственная таблица, в которую backend пишет — `leads`.

Это разделение позволяет backend и парсеру разрабатываться параллельно: их единственная точка контакта — схема таблиц `products`/`banks` и инварианты, гарантированные парсером (см. `parser.md` §10).

---

## 2. Эндпоинты (обзор)

| Метод | Путь | Назначение | Пишет в БД |
|---|---|---|---|
| GET | `/api/products` | Список активных продуктов с фильтрами, сортировкой, пагинацией | нет |
| GET | `/api/products/{id}` | Карточка одного продукта | нет |
| GET | `/api/banks` | Список активных банков | нет |
| POST | `/api/leads` | Приём заявки → запись + email | `leads` |

Полные контракты (query-параметры, тела, коды, JSON-примеры) — в `docs/api/contracts.md`. Этот файл описывает бизнес-правила за ними.

## 3. Бизнес-правила (инварианты API)

### 3.1. Видимость продуктов (жёсткий инвариант)

API отдаёт продукт ТОЛЬКО если выполнены ОБА условия:

- `products.status = 'active'`
- банк продукта (`banks.id = products.bank_id`) имеет `banks.status = 'active'`

Следствия:
- Продукт со `status ∈ {draft, hidden, outdated}` НИКОГДА не виден через API (ни в списке, ни по прямому `GET /api/products/{id}` — для скрытого id отдаётся `404`).
- Если банк `inactive`, все его продукты исчезают из выдачи разом (User Story 27), даже если сами продукты `active`.
- Это правило применяется на уровне базового запроса (global scope / общий билдер), а не точечно в каждом методе — чтобы исключить случайную утечку скрытых данных.

### 3.2. Заявки (leads)

- `consent` обязателен и должен быть строго `true`. Иначе — `422` (см. §4). Заявка без согласия НЕ сохраняется и email НЕ отправляется (User Story 17).
- При валидной заявке:
  1. создаётся запись в `leads` (`product_id`, `bank_id`, `full_name`, `phone`, `consent=true`, `created_at`);
  2. `bank_id` берётся НА СЕРВЕРЕ из `products.bank_id` по переданному `product_id` — клиент его не присылает и не может подменить (инвариант целостности);
  3. отправляется email на `banks.email` соответствующего банка через Laravel Mail.
- **`is_partner` НЕ влияет на маршрутизацию заявок.** Все заявки идут на `banks.email` одинаково, независимо от партнёрского статуса (PRD §6, User Story 18). Поле хранится для будущей дифференциации, но в MVP в логике leads не участвует.
- `product_id` должен ссылаться на ВИДИМЫЙ продукт (active + банк active). Заявка на скрытый/несуществующий продукт → `422` (поле `product_id` невалидно). Это предотвращает лиды на устаревшие предложения.

### 3.3. Порядок «запись → email» и устойчивость

- Запись в `leads` — первична и обязательна. Если запись прошла, заявка считается принятой (`201`), даже если последующая отправка email завершилась сбоем у почтового провайдера.
- Отправка email выполняется так, чтобы сбой почты не терял лид и не отдавал клиенту ошибку: рекомендуется постановка письма в очередь (queue) после успешной записи. Контракт наружу: `201` означает «лид сохранён», доставка email — best-effort с ретраями на уровне очереди.
- Адрес получателя — `banks.email`. Если у банка не задан email (граничный случай данных) — лид всё равно сохраняется, ошибка доставки логируется; клиенту отдаётся `201` (потеря лида недопустима).

## 4. Формат ошибок (единый, замороженный)

Все ошибки следуют единому формату. Backend использует стандартный механизм валидации Laravel.

### 4.1. Ошибки валидации — `422 Unprocessable Entity`

Стандартный ответ Laravel-валидатора:

```json
{
  "message": "The given data was invalid.",
  "errors": {
    "consent": ["Поле согласия обязательно и должно быть принято."],
    "phone": ["Телефон обязателен."]
  }
}
```

- Ключи в `errors` — имена полей запроса.
- Значение — массив человекочитаемых сообщений (локализованных, см. §6).
- Это применяется к `POST /api/leads` (тело) и к невалидным query-параметрам, где это уместно.

### 4.2. Не найдено — `404 Not Found`

Для `GET /api/products/{id}` и `GET` несуществующих ресурсов, а также для НЕвидимых продуктов (см. §3.1):

```json
{ "message": "Resource not found." }
```

### 4.3. Прочее

- `405` — метод не поддержан на ресурсе.
- `500` — внутренняя ошибка; тело НЕ раскрывает деталей реализации/стек.

> Семантика статусов едина по всем эндпоинтам (принцип consistent error semantics): `422` — семантически невалидные данные; `404` — нет ресурса/скрыт; `201` — создано; `200` — успешное чтение. Не смешивать (никаких «200 с error внутри»).

## 5. Валидация и фильтрация

### 5.1. Валидация входа (на границе)

- Тело `POST /api/leads` валидируется Form Request:
  - `full_name`: required, string, разумная длина (например 2–255).
  - `phone`: required, string, формат телефона (терпимый к таджикским форматам; нормализация на сервере).
  - `product_id`: required, integer, exists среди ВИДИМЫХ продуктов (см. §3.2).
  - `consent`: required, boolean, **должен быть `true`** (правило `accepted` или эквивалент). Любое значение ≠ true → `422`.
- Query-параметры `GET /api/products` валидируются на тип/диапазон/допустимые значения enum (см. contracts.md). Невалидный enum (например `currency=GBP`) → `422` либо игнор с дефолтом — **решение: невалидный enum → `422`**, чтобы фронт не молча получал не то, что просил.

### 5.2. Фильтры — через Query Builder (без raw SQL)

Все фильтры реализуются построением запроса Eloquent/Query Builder. Raw SQL запрещён (PRD: «фильтры через Query Builder, без raw SQL»). Поддерживаемые фильтры (детали значений — contracts.md):

| Параметр | Семантика фильтра |
|---|---|
| `category` | `products.category = ?` (`credit`/`deposit`) |
| `currency` | `products.currency = ?` (`TJS`/`USD`/`EUR`) |
| `amount_min` / `amount_max` | пересечение запрошенного диапазона суммы с `[amount_min, amount_max]` продукта (см. §5.3) |
| `term_min` / `term_max` | пересечение запрошенного диапазона срока с `[term_min, term_max]` продукта |
| `rate_min` / `rate_max` | фильтр по ставке; семантика тарифной сетки — §5.4 |
| `features[]` | продукт обладает ВСЕМИ перечисленными признаками (`features` jsonb содержит `true` по каждому ключу) |
| `sort` | сортировка (см. §5.5) |
| `page` / `per_page` | пагинация |

### 5.3. Семантика диапазонных фильтров (пересечение, не вложение)

- Продукт удовлетворяет фильтру по сумме, если запрошенный пользователем диапазон **пересекается** с допустимым диапазоном продукта:
  - продукт проходит, если `product.amount_min <= requested_amount_max` И (`product.amount_max IS NULL` ИЛИ `product.amount_max >= requested_amount_min`).
- Если задан только `amount_min` — фильтр «продукт доступен хотя бы для этой суммы»: `product.amount_max IS NULL OR product.amount_max >= amount_min`, и `product.amount_min <= amount_min` не требуется обязательным (продукт с большим минимумом тоже может быть интересен — **решение: используем пересечение по доступности**: продукт проходит, если запрошенная сумма попадает в его диапазон `[amount_min, amount_max]`, где `amount_max=NULL` трактуется как +∞).
- Аналогично для срока с учётом `term_max IS NULL` = +∞.
- `NULL` верхней границы (`amount_max`/`term_max`) трактуется как «без верхнего предела» во всех сравнениях.

### 5.4. Фильтр по ставке с учётом тарифной сетки (ключевое решение)

Ставка продукта многоуровневая (зависит от срока×суммы×валюты — `parser.md` §4). Контракт фильтрации:

- Каждый продукт имеет **агрегаты** `rate_min`/`rate_max` (минимум и максимум ставки по всем тирам, гарантированы парсером, см. `parser.md` §5.1).
- **Базовый фильтр `rate_min`/`rate_max` работает по агрегатам:** продукт проходит, если его диапазон ставок `[product.rate_min, product.rate_max]` **пересекается** с запрошенным `[rate_min, rate_max]`. Это даёт быстрый и предсказуемый фильтр без захода в jsonb-сетку.
- **Точное совпадение по ячейке сетки (term×amount×currency):** когда фронт присылает одновременно `currency` + `amount_*` + `term_*` + `rate_*`, backend дополнительно может сузить выдачу до продуктов, у которых СУЩЕСТВУЕТ тир, попадающий в заданные сумму/срок/валюту и удовлетворяющий ставке. Семантика (фиксируется):
  - продукт проходит «точный» режим, если существует `rate_tier`, у которого:
    `tier.currency = currency` И запрошенная сумма ∈ `[tier.amount_from, tier.amount_to]` И запрошенный срок ∈ `[tier.term_from, tier.term_to]` И `tier.rate` ∈ `[rate_min, rate_max]`.
  - Для этого backend читает `rate_tiers` (jsonb или связанная таблица — см. схему БД) и применяет фильтр по тиру через Query Builder (jsonb-операторы PostgreSQL допускаются как часть билдера, это не «raw SQL» в смысле сырых строк-запросов).
- **Что отдаётся клиенту:** в ответе продукт всегда содержит и агрегаты (`rate_min`/`rate_max`), и полную сетку `rate_tiers` (см. contracts.md). Клиент сам показывает релевантную ячейку (например подсвечивает ставку для введённых пользователем суммы/срока). Backend не «сплющивает» сетку до одного числа.

> Резюме контракта по ставке: **фронт запрашивает диапазоном** (`rate_min`/`rate_max`, опционально вместе с currency/amount/term для точного режима); **backend фильтрует по агрегатам, а при заданных amount+term+currency — дополнительно по существованию подходящего тира; отдаёт всегда и агрегаты, и полную сетку.**

### 5.5. Сортировка

- `sort` принимает значения вида `field` или `-field` (минус = по убыванию), допустимые поля: `rate_min`, `rate_max`, `amount_min`, `term_min`, `created_at`.
- По умолчанию — по ставке (`rate_min` по возрастанию, т.е. «лучшие сверху» для депозитов — уточняется UX; технически дефолт фиксируется в contracts.md).
- Недопустимое поле сортировки → `422`.

## 6. Мультиязычность и мультивалютность в ответах

- Продукты и банки отдаются С ОБОИМИ языковыми полями (`name_ru`, `name_tg` и т.п.) — выбор языка делает фронт по активному locale (`ru`/`tg`). Backend НЕ выбирает язык за клиента и НЕ скрывает одно из полей.
- `name_tg` может быть `null` (парсер не нашёл перевод) — это валидное значение; фронт делает fallback на `ru`.
- Сообщения об ошибках валидации (`422.errors`) локализуются по `Accept-Language` (`ru`/`tg`); дефолт — `ru`.
- Валюта продукта (`currency`) — одна на запись (`TJS`/`USD`/`EUR`), см. решение в `parser.md` §4. Backend не конвертирует валюты (конвертация вне MVP).

## 7. Граничные случаи

| Случай | Поведение |
|---|---|
| `GET /api/products/{id}` для скрытого/несуществующего продукта | `404` |
| Банк продукта стал `inactive` | продукт исчезает из всех выдач; прямой запрос → `404` |
| `POST /api/leads` с `consent=false`/отсутствует | `422`, не сохранён, email не отправлен |
| `product_id` ссылается на скрытый продукт | `422` (поле `product_id`) |
| Email банка пуст/невалиден | лид сохранён, `201`; ошибка доставки логируется |
| Сбой почтового провайдера | лид сохранён, `201`; письмо ретраится очередью |
| `currency`/`category`/`sort` с недопустимым значением | `422` |
| Пустой результат фильтрации | `200` с пустым `data` и корректной пагинацией |
| Продукт без `name_tg` | отдаётся `name_tg: null` |
| Дробная/строковая `amount_*` в query | валидация типа; невалидно → `422` |

## 8. Тестируемость (внешнее поведение)

Feature-тесты с `RefreshDatabase` (PRD Testing Decisions):
- `GET /api/products` возвращает только `status=active` + банк `active`.
- Фильтры (category, currency, amount, term, rate, features) корректно сужают выборку; фильтр по ставке учитывает агрегаты и тиры (§5.4).
- `POST /api/leads` без `consent=true` → `422`.
- `POST /api/leads` валидный → запись в `leads` + `Mail::fake()` перехватывает письмо на `banks.email`.
- `is_partner` не меняет адрес доставки (письмо всегда на `banks.email`).
- Скрытый продукт → `404`; лид на скрытый продукт → `422`.

## 9. Контракт с другими модулями

- **С парсером:** общая БД; backend полагается на инварианты валидности (`parser.md` §5) и семантику `status`. Не пишет в `products`.
- **С фронтом:** строго по `docs/api/contracts.md` — это ЗАМОРОЖЕННЫЙ контракт. Изменения только аддитивные (новые опциональные поля/параметры), без breaking changes (принцип «addition over modification»).
- **С почтой:** Laravel Mail + очередь; адрес — всегда `banks.email`.

## 10. Критерии готовности

- [ ] Каталог отдаёт только `active`+`active` (продукт+банк); скрытое недоступно даже по id.
- [ ] Все фильтры через Query Builder, без сырого SQL; ставка фильтруется по §5.4.
- [ ] `POST /api/leads`: `consent!=true` → `422`; валидный → запись + email на `banks.email`.
- [ ] `bank_id` лида определяется сервером по `product_id`.
- [ ] `is_partner` не влияет на доставку.
- [ ] Единый формат ошибок (`422`+`errors{}`, `404`); статусы не смешиваются.
- [ ] Ответы содержат оба языковых поля; валюта одна на запись.
- [ ] Ответы соответствуют `docs/api/contracts.md` байт-в-байт по структуре.
