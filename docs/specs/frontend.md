# Спецификация: Frontend (Vue.js)

**Модуль:** `frontend/`
**Технологии:** Vue.js + Pinia (state) + Vue I18n (локализация) + Vue Router.
**Назначение:** витрина сравнения банковских продуктов — каталог, карточка, сравнение, калькулятор, форма заявки.
**Статус:** контракт фундамента. Описывает ВНЕШНЕЕ поведение (страницы, состояния, инварианты UX, обращения к API), не детали компонентной реализации. Источник данных — REST API из `docs/api/contracts.md` (замороженный контракт). Дизайн — палитра eskhata.com (см. PRD §«Цветовая палитра»).

---

## 1. Назначение и границы

- **Делает:** рендерит каталог/карточки/сравнение, фильтрует и сортирует (через API), считает калькулятор на клиенте, собирает и отправляет заявки, переключает язык.
- **НЕ делает:** не хранит бизнес-данные локально как источник истины (источник — API), не считает ставки/суммы за парсер, не имеет авторизации/админки (вне MVP), не конвертирует валюты.

---

## 2. Страницы и маршруты

| Маршрут | Страница | Источник данных |
|---|---|---|
| `/credits` | Каталог кредитов | `GET /api/products?category=credit&...` |
| `/deposits` | Каталог депозитов | `GET /api/products?category=deposit&...` |
| `/products/:id` | Карточка продукта | `GET /api/products/{id}` |
| `/compare` | Сравнение (до 3–4 продуктов) | из Pinia (id-набор) + при необходимости догрузка по API |
| (модал/страница) | Форма заявки | `POST /api/leads` |

Каталоги кредитов и депозитов — одна логика, разные значения `category`.

### 2.1. Каталог (кредиты / депозиты)

- Список карточек продуктов с пагинацией (по `pagination` из API).
- Панель фильтров (см. §3) и контрол сортировки (`sort`).
- Каждая карточка: банк (`name_ru`/`name_tg` по locale), категория, валюта, диапазон ставок (`rate_min–rate_max`), диапазоны суммы/срока, ключевые `features`-бейджи, кнопки «Подробнее», «В сравнение», «Оставить заявку».
- Пустой результат фильтрации → дружелюбное «ничего не найдено», без ошибки.
- Состояния: loading / loaded / empty / error (ошибка сети → ретрай-кнопка).

### 2.2. Карточка продукта `/products/:id`

- Полные данные продукта: банк, мультиязычные название/описание, валюта, агрегаты ставок и **полная тарифная сетка `rate_tiers`** (термин×сумма×валюта) в виде таблицы.
- Встроенный калькулятор (см. §5), преднастроенный данными продукта.
- Кнопки «В сравнение» и «Оставить заявку».
- `404` от API → страница «продукт не найден» (продукт мог стать `hidden`/банк `inactive`).

### 2.3. Сравнение `/compare`

- До **3–4** продуктов одновременно бок о бок (лимит — инвариант §4).
- Колонки = продукты, строки = атрибуты (банк, ставки, сумма, срок, features). Различия визуально подсвечиваются.
- Набор сравнения хранится в Pinia (§4) и переживает навигацию между страницами.
- Можно удалить продукт из сравнения и перейти в карточку.

### 2.4. Форма заявки

- Поля: `full_name`, `phone`, (скрытое) `product_id`, `consent` (чекбокс).
- Чекбокс `consent` со ссылкой на политику обработки персональных данных.
- См. §6 — правила consent и submit.

## 3. Фильтры и сортировка (контракт с API)

Фронт транслирует UI-контролы в query-параметры `GET /api/products` (точные имена/форматы — contracts.md):

| UI-контрол | Параметр(ы) |
|---|---|
| Категория (определяется маршрутом) | `category` |
| Валюта (TJS/USD/EUR), single-select | `currency` |
| Сумма (слайдер/поля от–до) | `amount_min`, `amount_max` |
| Срок (от–до, месяцы) | `term_min`, `term_max` |
| Ставка (от–до, %) | `rate_min`, `rate_max` |
| Признаки (чекбоксы: онлайн, без поручителя, капитализация, пополнение) | `features[]` |
| Сортировка | `sort` (например `rate_min`, `-rate_max`) |
| Страница | `page`, `per_page` |

### 3.1. Запрос ставки с учётом тарифной сетки (ключевое поведение)

- Базовый фильтр по ставке фронт всегда шлёт диапазоном `rate_min`/`rate_max`.
- Если пользователь задал ОДНОВРЕМЕННО валюту + сумму + срок, фронт может включить «точный режим», отправив `currency` + `amount_*` + `term_*` + `rate_*` вместе — backend тогда фильтрует по существованию подходящей ячейки сетки (см. `backend.md` §5.4).
- **Отображение:** фронт получает от API и агрегаты (`rate_min`/`rate_max`), и полную `rate_tiers`. На карточке/в каталоге показывается диапазон; если пользователь ввёл конкретные сумму+срок+валюту — фронт подсвечивает соответствующую ячейку сетки (ставку именно для его параметров), вычисляя её на клиенте из `rate_tiers`. Backend ставку «за пользователя» не выбирает.

### 3.2. Поведение

- Изменение фильтра → новый запрос к API (debounce на слайдерах/вводе), URL-синхронизация query-параметров (шарабельные ссылки, кнопка «назад» работает).
- Невалидную комбинацию фронт не формирует (например `amount_min > amount_max` блокируется в UI до отправки).

## 4. Состояние сравнения (Pinia)

- Store хранит набор id выбранных для сравнения продуктов + закэшированные краткие данные карточек.
- **Инвариант лимита:** не более 4 продуктов (минимум 2 для осмысленного сравнения). При попытке добавить сверх лимита — UI блокирует и подсказывает убрать один.
- Добавление/удаление доступно из каталога и карточки; индикатор количества — в шапке/«корзине сравнения».
- Состояние переживает навигацию (в рамках сессии). Персистентность между сессиями — опционально (localStorage), не обязательна для MVP.

## 5. Калькулятор (полностью на клиенте)

Вычисления выполняются на клиенте в реальном времени (без кнопки, реактивно при изменении входов — User Story 13). Данные берутся из карточки продукта (предзаполнение) или вводятся вручную. Калькулятор НЕ обращается к API и не парсит ставки.

### 5.1. Кредит — аннуитетный платёж

Вход: сумма `P`, срок `n` (месяцев), годовая ставка `r%`.
Месячная ставка `i = r / 100 / 12`.

- Если `i > 0`: ежемесячный платёж `A = P * i * (1+i)^n / ((1+i)^n − 1)`.
- Если `i = 0` (граничный случай): `A = P / n`.
- Производные: переплата = `A*n − P`, итог к выплате = `A*n`.

### 5.2. Депозит — простые / сложные проценты

Вход: сумма `P`, срок `n` (месяцев), годовая ставка `r%`, флаг капитализации, периодичность капитализации (если есть).

- **Простые проценты** (без капитализации): доход = `P * (r/100) * (n/12)`; итог = `P + доход`.
- **Сложные проценты** (капитализация, `m` начислений в год): итог = `P * (1 + (r/100)/m)^(m * n/12)`; доход = итог − `P`.
- Капитализация включается, только если у продукта `features.capitalization = true` (иначе контрол скрыт/выключен).

### 5.3. Инварианты калькулятора

- Все входы валидируются: `P > 0`, `n >= 1`, `0 < r <= 100`. Невалидный ввод → результат не считается, показывается подсказка.
- Результаты форматируются в валюте продукта (`currency`), с учётом locale-форматирования чисел (`ru`/`tg`).
- Округление денежных значений — до 2 знаков для отображения; промежуточные вычисления — без преждевременного округления.
- Если у продукта тарифная сетка, для введённых пользователем суммы+срока подставляется ставка из соответствующего тира `rate_tiers` (а не агрегат), чтобы расчёт был точным.

## 6. Форма заявки и consent (жёсткий инвариант)

- Поля: `full_name` (required), `phone` (required), `product_id` (из контекста продукта, скрытое), `consent` (чекбокс).
- **Кнопка submit `disabled`, пока `consent` не отмечен** (PRD: «кнопка submit disabled без галочки»; User Story 17). Это первичная защита на клиенте.
- Чекбокс сопровождается текстом со ссылкой на политику обработки персональных данных.
- Клиентская валидация (непустые ФИО/телефон, формат телефона) до отправки; но **сервер — источник истины**: фронт обязан корректно обрабатывать `422` от `POST /api/leads`.

### 6.1. Обработка ответов `POST /api/leads`

| Ответ | Поведение фронта |
|---|---|
| `201` | сообщение об успехе («Заявка отправлена»), форма сбрасывается/закрывается |
| `422` | разбор `errors{}`: показать сообщения под соответствующими полями (`full_name`/`phone`/`product_id`/`consent`) |
| `404` (продукт исчез) | сообщить, что продукт более недоступен |
| сеть/`500` | общая ошибка + возможность повторить отправку, данные формы сохраняются |

- `consent` отправляется как `true`; фронт никогда не отправляет форму без отмеченного чекбокса.

## 7. Мультиязычность (Vue I18n)

- Два locale: `ru` (дефолт) и `tg`. Переключатель в шапке; выбор сохраняется (localStorage) и переживает перезагрузку.
- **UI-строки** — из словарей i18n (`ru.json`/`tg.json`).
- **Данные продуктов/банков** — приходят с обоими полями (`name_ru`/`name_tg`, `description_ru`/`description_tg`); отображается поле по активному locale.
- **Fallback:** если поле текущего locale пустое/`null` (например `name_tg = null`), показывается `*_ru`. Обратный fallback не требуется (`ru` всегда есть).
- Форматирование чисел/валют — по locale.
- Заголовок `Accept-Language` выставляется по активному locale (чтобы серверные сообщения `422` приходили локализованными).

---

## 8. Обращение к API (контракт)

- Источник данных — строго `docs/api/contracts.md`. Фронт не предполагает недокументированного поведения (принцип: не зависеть от незадокументированных квирков).
- Все языковые поля и `rate_tiers` приходят в ответе — фронт сам решает, что показать.
- Ошибки обрабатываются по единому формату (`422`+`errors{}`, `404`, `500`) — см. `backend.md` §4.
- Пагинация рендерится по объекту `pagination`.

## 9. Дизайн-система

- Палитра — по eskhata.com (PRD §«Цветовая палитра»): Primary `#0050C8`, danger `#EA3C4A` (ошибки валидации), accent-green `#598E32` (успех/успешная заявка) и т.д. Используются CSS-переменные из PRD.

## 10. Граничные случаи

| Случай | Поведение |
|---|---|
| `name_tg = null` | fallback на `name_ru` |
| Пустой каталог после фильтра | «ничего не найдено», не ошибка |
| Продукт стал `hidden` (открыт по прямой ссылке) | `404` → «продукт недоступен» |
| Попытка добавить 5-й продукт в сравнение | блок + подсказка |
| `consent` не отмечен | submit `disabled` |
| `422` от сервера | сообщения под полями |
| Сбой сети при отправке заявки | данные формы сохранены, кнопка «повторить» |
| Калькулятор: ставка 0 (кредит) | формула `P/n` |
| Калькулятор: продукт с сеткой | ставка берётся из тира под введённые сумму/срок |
| Переключение языка | UI и отображаемые поля данных меняются мгновенно |

## 11. Тестируемость (внешнее поведение)

- Юнит (Vitest + Vue Test Utils): калькулятор (числа при разных входах), форма (`disabled` без consent / `enabled` с consent), i18n (переключение locale меняет текст и применяет fallback).
- E2E (Playwright): применение фильтров обновляет список; добавление в сравнение → страница сравнения бок о бок; заполнение и отправка формы → сообщение об успехе.

## 12. Критерии готовности

- [ ] Каталоги кредитов/депозитов с фильтрами, сортировкой, пагинацией по API.
- [ ] Карточка с полной тарифной сеткой и калькулятором.
- [ ] Сравнение до 3–4 продуктов через Pinia, переживает навигацию.
- [ ] Калькулятор на клиенте, реактивный; аннуитет/простые/сложные проценты по §5.
- [ ] consent блокирует submit; `422` корректно разбирается.
- [ ] i18n ru/tg с fallback на ru; оба языковых поля рендерятся по locale.
- [ ] Все обращения к API соответствуют contracts.md; единая обработка ошибок.
