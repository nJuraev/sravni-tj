# Спецификация: Парсер продуктов (Go)

**Модуль:** `parser/`
**Язык:** Go
**Запуск:** по крону (периодически), один процесс на запуск.
**Статус:** контракт фундамента. Описывает ВНЕШНЕЕ поведение модуля — что на входе, что на выходе в БД, инварианты и ошибки. НЕ описывает внутреннюю реализацию (структуры, имена функций).

---

## 1. Назначение и границы

Парсер — единственный писатель в таблицы `products` и (опционально) `parser_runs`. Он автоматически собирает актуальные условия по кредитам и депозитам с сайтов банков Таджикистана, приводит их к структурированному виду через AI и записывает в общую PostgreSQL.

**Границы ответственности:**

- **Делает:** читает задачи из `bank_source_urls`, скрейпит страницы → Markdown, извлекает структуру через AI, валидирует, пишет в `products`.
- **НЕ делает:** не отдаёт HTTP API (это backend), не управляет статусами банков, не отправляет заявки, не делает ручное ревью. Не пишет в `banks`, `bank_source_urls`, `leads`.
- **Единственный писатель `products`:** backend и фронт только читают. Это инвариант разделения, позволяющий парсеру и backend разрабатываться параллельно без конфликтов по данным.

## 2. Вход

### 2.1. Источник задач — таблица `bank_source_urls`

При старте парсер выполняет выборку активных источников:

```
SELECT * FROM bank_source_urls WHERE is_active = true
```

Каждая строка — одна **задача парсинга** со следующими значимыми полями:

| Поле | Тип | Значение для парсера |
|---|---|---|
| `id` | int | идентификатор источника; записывается в `products.source_url_id` и `parser_runs.bank_source_url_id` |
| `bank_id` | int | банк, к которому привязаны продукты |
| `category` | enum(`credit`,`deposit`) | категория продуктов на этой странице |
| `url` | string | URL страницы для скрейпинга |
| `is_active` | bool | только `true` попадают в обработку |

**Мультидоменность (граничный случай, обязателен):** один банк может иметь несколько строк `bank_source_urls` с разными URL и категориями (например, `https://deposit.dc.tj` с `category=deposit` и `https://credit.dc.tj` с `category=credit`). Парсер обрабатывает каждую строку независимо как отдельную задачу — никакой специальной логики «склейки доменов» не требуется. Один банк = N задач = N независимых результатов.

### 2.2. Конфигурация (через env)

| Переменная | Тип | По умолчанию | Назначение |
|---|---|---|---|
| `PARSER_DEBUG_LOG` | bool | `false` | Включает запись метаданных каждого запуска в `parser_runs`. См. §6. |
| `DATABASE_URL` | string | — | Подключение к PostgreSQL (общая БД). |
| `SCRAPER_PROVIDER` | enum(`firecrawl`,`jina`) | — | Провайдер HTML→Markdown. |
| `SCRAPER_API_KEY` | string (секрет) | — | Ключ скрейпера. Читается из env, не логируется. |
| `AI_PROVIDER` | enum(`gemini`,`qwen`) | — | AI-модель для извлечения. |
| `AI_API_KEY` | string (секрет) | — | Ключ AI. Читается из env, не логируется. |
| `PARSER_CONCURRENCY` | int | `1` | Максимум задач, обрабатываемых параллельно. |
| `PARSER_HTTP_TIMEOUT_SEC` | int | `60` | Таймаут на скрейп одной страницы. |
| `PARSER_AI_TIMEOUT_SEC` | int | `120` | Таймаут на один вызов AI. |

> Секреты (`*_API_KEY`, `DATABASE_URL`) читаются только из окружения и НИКОГДА не попадают в логи и в `parser_runs.ai_raw_response`.

## 3. Pipeline (конвейер обработки одной задачи)

```
[bank_source_urls row]
        │
        ▼
(1) SCRAPE      url → HTML → Markdown    (Firecrawl / Jina)
        │
        ▼
(2) EXTRACT     Markdown → AI structured output → typed JSON по схеме §4
        │
        ▼
(3) VALIDATE    проверка диапазонов и инвариантов §5
        │
        ▼
(4) PERSIST     UPSERT в products (атомарно на задачу)
        │
        ▼
(5) LOG         запись parser_runs (только если PARSER_DEBUG_LOG=true)
```

### Этап 1 — Scrape (HTML → Markdown)

- Загружает `url` через выбранный `SCRAPER_PROVIDER`, получает Markdown-представление страницы.
- Markdown снижает хрупкость к изменению вёрстки (по сравнению с CSS-селекторами).
- При неудаче (сеть, таймаут, не-2xx, пустой ответ) — задача завершается ошибкой этапа (см. §7). Этапы 2–4 не выполняются.

### Этап 2 — Extract (Markdown → структурированный JSON через AI)

- AI вызывается **строго в режиме structured output / function calling** с фиксированной JSON-схемой (§4). Свободный текст от модели не принимается.
- Цель — анти-галлюцинации: модель обязана вернуть JSON по схеме либо ничего (пустой массив продуктов).
- Сырой ответ модели сохраняется в памяти для возможной записи в `parser_runs` (этап 5).

### Этап 3 — Validate

- Каждый продукт из ответа AI прогоняется через инварианты §5.
- **Невалидный продукт отбраковывается, но не валит всю задачу** (см. §5.4 — политика частичной отбраковки). Валидные продукты этой же задачи записываются.

### Этап 4 — Persist (запись в products)

- Запись выполняется как UPSERT (см. §8 — стратегия идемпотентности).
- `products.status` при записи парсером — `active` (продукт сразу виден), если иное не предписано политикой банка/источника. Изменение статуса на `hidden`/`outdated` — прерогатива администратора БД, парсер его не трогает при последующих запусках, КРОМЕ перевода исчезнувших продуктов в `outdated` (см. §8.3).
- `parsed_at` проставляется текущим временем записи.

### Этап 5 — Log (метаданные)

См. §6.

## 4. Контракт AI-схемы (structured output)

AI обязан вернуть объект с массивом продуктов. Схема — стабильный контракт между парсером и AI-провайдером; смена провайдера (Gemini ↔ Qwen) не должна менять схему.

**Концептуальная форма ответа (не код, описание полей):**

```
{
  "products": [
    {
      "name_ru":  string,            // название продукта на русском (обязательно)
      "name_tg":  string,            // название на таджикском; если на странице нет — null
      "category": "credit"|"deposit",// должно совпадать с category задачи (см. §5.3)
      "currency": "TJS"|"USD"|"EUR", // валюта продукта
      "rate_min": number,            // мин. годовая ставка, %  (см. §5.1)
      "rate_max": number,            // макс. годовая ставка, % (см. §5.1)
      "amount_min": number | null,   // мин. сумма; null = не указана (см. §5.2)
      "amount_max": number | null,   // макс. сумма; null = без верхнего предела
      "term_min": integer,           // мин. срок, месяцев (см. §5.2)
      "term_max": integer | null,    // макс. срок, месяцев; null = бессрочно/до востребования
      "rate_tiers": [                // тарифная сетка: ставка зависит от срока×суммы×валюты
        {
          "currency":   "TJS"|"USD"|"EUR",
          "amount_from": number,
          "amount_to":   number | null,
          "term_from":   integer,        // месяцев
          "term_to":     integer | null, // месяцев
          "rate":        number          // %, для данной ячейки сетки
        }
      ],
      "features": {                  // признаки; отсутствующие = false/null
        "online_application": boolean,  // онлайн-оформление
        "no_guarantor":       boolean,  // без поручителя
        "capitalization":     boolean,  // капитализация процентов (депозит)
        "replenishment":      boolean   // возможность пополнения (депозит)
      }
    }
  ]
}
```

**Мультиязычность:** `name_ru` обязателен; `name_tg` может быть `null`, если перевод отсутствует на источнике — backend/фронт обязаны корректно обрабатывать `null` (fallback на `name_ru`). Парсер НЕ переводит сам.

**Мультивалютность и тарифные сетки (ключевой граничный случай — User Story 25):**
- Простой случай: продукт в одной валюте с единым диапазоном ставок → `rate_min`/`rate_max` заполнены, `rate_tiers` может содержать одну запись или агрегировать диапазон.
- Сложный случай: ставка зависит от суммы И/ИЛИ срока И/ИЛИ валюты → детальная сетка в `rate_tiers`. `rate_min`/`rate_max` в этом случае = агрегаты (минимум и максимум `rate` по всем тирам) — они нужны для быстрой фильтрации/сортировки на backend, а `rate_tiers` — для точного отображения и фильтрации «по конкретной ячейке».
- Если у банка один продукт представлен в нескольких валютах с разными ставками — это допускается выразить ИЛИ как несколько записей `products` (по одной на валюту), ИЛИ как один продукт с `rate_tiers`, различающимися по `currency`. **Решение (фиксируется здесь): одна запись `products` = одна валюта (`products.currency`).** Мультивалютный продукт банка разбивается парсером на N записей products — по одной на валюту. `rate_tiers` внутри одной записи содержат тиры только её валюты. Это упрощает фильтр `currency` на backend (см. contracts.md).

## 5. Инварианты валидации

Валидация — последний барьер перед записью в БД. Нарушение инварианта = продукт не записывается (отбраковка, §5.4).

### 5.1. Ставки

- `0 < rate <= 100` для каждого значения ставки (`rate_min`, `rate_max`, каждый `rate_tiers[].rate`). Граница: `0` недопустима (ставка 0% считается ошибкой извлечения), `100` — допустима.
- `rate_min <= rate_max`.
- Для тарифной сетки: `min(rate_tiers[].rate) == rate_min` и `max(rate_tiers[].rate) == rate_max` (агрегаты согласованы с сеткой). При рассогласовании — агрегаты пересчитываются из сетки (сетка — источник истины), рассогласование логируется как warning.

### 5.2. Суммы и сроки

- `amount_min`: `null` (не указана) либо `> 0`. `amount_max` либо `null`, либо `>= amount_min`.
- `term_min >= 1` (месяц). `term_max` либо `null`, либо `>= term_min`.
- В каждом тире: `amount_from > 0`, `amount_to` (`null` | `>= amount_from`); `term_from >= 1`, `term_to` (`null` | `>= term_from`).

### 5.3. Категория и валюта

- `category` продукта обязан совпадать с `category` задачи (`bank_source_urls.category`). Если AI вернул кредит на странице депозитов — продукт отбраковывается (граничный случай: страница с микс-контентом или ошибка извлечения).
- `currency ∈ {TJS, USD, EUR}`. Иные валюты — отбраковка.

### 5.4. Политика частичной отбраковки

- Невалидный продукт **не валит всю задачу**. Валидные продукты той же задачи записываются.
- Если **все** продукты задачи отбракованы (или AI вернул пустой массив) — задача считается завершённой с пустым результатом; существующие в БД продукты этого источника НЕ удаляются автоматически на этом запуске (см. §8.3 про устаревание), но событие фиксируется в логах/`parser_runs` со статусом `empty`.

### 5.5. Целостность мультиязычных полей

- `name_ru` непустой после trim — обязателен. Пустой `name_ru` = отбраковка продукта.
- `name_tg` = `null` допустимо.

## 6. Конфиг-флаг логирования `PARSER_DEBUG_LOG`

| `PARSER_DEBUG_LOG` | Поведение |
|---|---|
| `false` (по умолчанию) | В `parser_runs` НИЧЕГО не пишется. Парсер работает «тихо». Обычные операционные логи (stdout) не относятся к `parser_runs`. |
| `true` | Для КАЖДОЙ задачи (каждой строки `bank_source_urls`) создаётся запись в `parser_runs`. |

**Запись `parser_runs` (при флаге `true`):**

| Поле | Содержимое |
|---|---|
| `bank_source_url_id` | id задачи |
| `started_at` | время начала обработки задачи |
| `finished_at` | время завершения (успех или ошибка) |
| `ai_raw_response` | сырой текст ответа AI-модели (до парсинга), для отладки галлюцинаций; **секреты не включаются** |
| `status` | `success` \| `empty` \| `scrape_error` \| `ai_error` \| `validation_error` \| `db_error` |
| `error_message` | человекочитаемое описание ошибки или `null` |

**Инвариант:** флаг влияет ТОЛЬКО на запись в `parser_runs`. Он НЕ меняет логику скрейпинга/извлечения/записи в `products`. То есть результат в `products` идентичен при `true` и `false`.

## 7. Ошибки и ретраи

### 7.1. Классификация ошибок (per-task)

| Класс | Этап | Пример | Поведение |
|---|---|---|---|
| `scrape_error` | 1 | таймаут, 4xx/5xx, пустой Markdown | задача провалена, ретрай (§7.2) |
| `ai_error` | 2 | таймаут AI, невалидный JSON относительно схемы, отказ модели | задача провалена, ретрай |
| `validation_error` | 3 | все продукты отбракованы по §5 | задача завершена со `status=empty`/`validation_error`, БЕЗ ретрая (ретрай не поможет — данные детерминированы) |
| `db_error` | 4 | сбой записи в PostgreSQL | задача провалена, ретрай |

### 7.2. Стратегия ретраев

- Ретраятся только транзиентные ошибки: `scrape_error`, `ai_error` (таймаут/сетевые/rate-limit), `db_error`.
- Экспоненциальный backoff: базовая задержка с ростом (например 1s → 2s → 4s), максимум 3 попытки на задачу.
- Rate-limit от AI/скрейпера (HTTP 429) уважается: backoff не короче `Retry-After`, если он отдан.
- `validation_error` НЕ ретраится (детерминированный результат).
- Исчерпание ретраев = задача помечается провалившейся; при `PARSER_DEBUG_LOG=true` — соответствующий `status` и `error_message` в `parser_runs`.

### 7.3. Изоляция задач

- **Падение одной задачи не останавливает остальные.** Каждая строка `bank_source_urls` обрабатывается изолированно. Сбой источника банка A не влияет на источник банка B.
- Процесс парсера завершается с ненулевым кодом, только если упал сам процесс (например, нет подключения к БД на старте). Частичные провалы отдельных задач = код выхода 0 (это нормальная операция; детали — в `parser_runs`/логах).

## 8. Идемпотентность и обновление данных

### 8.1. UPSERT

Повторный запуск парсера не должен плодить дубли. Запись — UPSERT по бизнес-ключу продукта.

- **Бизнес-ключ продукта (для дедупликации в рамках одного источника):** `(source_url_id, category, currency, name_ru)`. Этот набор стабильно идентифицирует продукт на повторных запусках.
- Совпадение по ключу → UPDATE существующей записи (ставки/суммы/сроки/features/`parsed_at`), сохраняя `id` и созданный администратором `status` (если он был вручную изменён — см. §8.2).
- Нет совпадения → INSERT с `status=active`.

### 8.2. Уважение администраторских статусов

- Если администратор вручную выставил продукту `status ∈ {hidden, draft, outdated}`, парсер при обновлении данных **не возвращает** `status` в `active` автоматически. Он обновляет числовые/контентные поля, но статус, выставленный вручную, не перетирает на `active`.
- Это инвариант управляемости: §26 PRD (администратор скрывает некорректные данные без деплоя), парсер не должен «воскрешать» скрытое.

### 8.3. Устаревание исчезнувших продуктов

- Если продукт ранее парсился из источника, но в текущем успешном запуске (`status=success`, непустой результат) больше не присутствует, парсер переводит его `status` → `outdated`. Это держит витрину актуальной без удаления исторических строк.
- При `status=empty`/ошибке задачи устаревание НЕ применяется (нельзя отличить «продуктов больше нет» от «не удалось спарсить»).

## 9. Граничные случаи (сводка)

| Случай | Ожидаемое поведение |
|---|---|
| Банк с несколькими URL/доменами | N независимых задач, по строке `bank_source_urls`; никакой склейки |
| Страница без продуктов / AI вернул `[]` | `status=empty`, существующие продукты не трогаются |
| Часть продуктов невалидна | валидные пишутся, невалидные отбраковываются |
| `name_tg` отсутствует на источнике | `name_tg = null`, продукт валиден |
| Мультивалютный продукт | разбивается на N записей `products` (по валюте) |
| Сложная сетка ставок | `rate_tiers` + согласованные агрегаты `rate_min`/`rate_max` |
| Ставка = 0% от AI | отбраковка (вероятная галлюцинация) |
| `amount_max`/`term_max` не указаны | `null` = без верхнего предела |
| Источник стал недоступен (404/timeout) | ретраи; при исчерпании — провал задачи, остальные продолжаются |
| Продукт исчез со страницы | перевод в `outdated` при успешном непустом запуске |
| Администратор скрыл продукт (`hidden`) | парсер обновляет данные, но не возвращает `active` |

## 10. Контракт с другими модулями

- **С БД (PostgreSQL):** парсер — единственный писатель `products`/`parser_runs`. Схема таблиц — общий контракт; см. CLAUDE.md и отдельную спеку схемы БД.
- **С backend (Laravel):** связь только через таблицу `products`. Backend полагается на инварианты §5 (валидность диапазонов) и на семантику `status`. Парсер гарантирует, что `status=active` означает «прошёл валидацию и виден».
- **С внешними сервисами (Firecrawl/Jina, Gemini/Qwen):** ответы третьих сторон считаются НЕдоверенными — AI-ответ принимается только через жёсткую schema-валидацию (§4, §5), скрейп-результат проверяется на непустоту.

## 11. Критерии готовности (success criteria)

- [ ] Парсер читает только `is_active=true` из `bank_source_urls`.
- [ ] AI вызывается строго в structured-output режиме по схеме §4.
- [ ] Все инварианты §5 проверяются до записи; нарушители отбраковываются без падения задачи.
- [ ] `PARSER_DEBUG_LOG=true` создаёт записи `parser_runs`; `false` — не создаёт; результат в `products` идентичен.
- [ ] Повторный запуск идемпотентен (нет дублей по бизнес-ключу §8.1).
- [ ] Мультидоменный банк парсится по всем своим источникам.
