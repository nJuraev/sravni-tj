# Техническое задание

## Агрегатор банковских продуктов Таджикистана (аналог Sravni.ru)

**Версия:** 1.0 (MVP)
**Дата:** 2026-06-04

---

## 1. Цель проекта

Веб-платформа для сравнения банковских продуктов Таджикистана (аналог sravni.ru).
Пользователь сравнивает кредиты и депозиты разных банков и может оставить заявку на
выбранный продукт. Данные о продуктах собираются автоматически парсером с сайтов банков.

---

## 2. Состав системы (монорепо)

```
sravni-tj/
├── parser/        # Go — парсер продуктов (по крону)
├── backend/       # PHP / Laravel — API + приём заявок
├── frontend/      # Vue.js — витрина сравнения + формы заявок
└── docker-compose.yml
```

| Компонент | Технология | Назначение |
|---|---|---|
| Парсер | Go | Сбор продуктов с сайтов банков по крону |
| БД | PostgreSQL | Хранилище продуктов, банков, заявок |
| Бэкенд | PHP / Laravel | REST API, приём и доставка заявок |
| Фронтенд | Vue.js | Витрина сравнения, фильтры, формы заявок |

Парсер пишет данные напрямую в общую БД PostgreSQL; Laravel читает из той же БД.

---

## 3. Парсер

- Парсит продукты с **конкретных сайтов банков** (до 20 банков:
  Эсхата, Алиф, Душанбе Сити и др.).
- **Продукты на MVP:** кредиты + депозиты.
- **Количество банков:** до 20 (точный список предоставляется заказчиком).
- **Подход к извлечению:** страница банка → Markdown (Firecrawl / Jina) →
  бесплатная AI-модель (Gemini / Qwen) извлекает структурированные данные →
  запись в PostgreSQL. AI обязан возвращать строго типизированный JSON по схеме
  (анти-галлюцинации для финансовых цифр).
- **Запуск по крону** — периодическое обновление данных.
- **Метаданные парсинга** (источник URL, время парсинга, сырой ответ AI, статус запуска)
  предварительно **не сохраняются**. Должен быть **конфиг-флаг**, включающий логирование
  этих данных (для отладки AI-извлечения при необходимости).
- **Особенности данных, которые парсер обязан учитывать:**
  - ставка зависит от срока, суммы и валюты;
  - мультивалютность: TJS, USD, EUR;
  - у банка может быть несколько доменов/сабдоменов (напр. DC: `deposit.dc.tj`, `credit.dc.tj`).

- Названия и описания продуктов и банков хранятся мультиязычно.

## 5. Сайт сравнения (ядро)

- Каталог продуктов по категориям: **кредиты, депозиты**.
- **Сравнение продуктов разных банков** бок о бок.
- **Фильтры:** сумма, срок, валюта, ставка, фичи (онлайн-оформление, без поручителя,
  капитализация, пополнение и т.п.).
- **Сортировка:** по ставке и другим параметрам.
- **Карточка продукта:** банк, ставка, условия, требования.
- **Калькулятор:** доход по вкладу / платёж по кредиту — вычисляется на лету,
  не парсится.
- **Без авторизации пользователей** на MVP.

---

## 6. Заявки (leads) и B2B-партнёрство

- Пользователь может **оставить заявку** на продукт (кредит / депозит) прямо на сайте.
- Заявка **отправляется в банк по email** — **всегда**, независимо от партнёрского статуса
  (email банка несложно найти публично).
- **Канал доставки заявки:** предварительно — **email** на адрес банка.
- Признак «банк-партнёр» хранится у банка как опциональный атрибут (для будущей
  приоритизации/интеграции), но на доставку заявки сейчас не влияет.
- Это основа монетизации (лидогенерация).

### Форма заявки
- Поля: ФИО, телефон (и минимально необходимые данные).
- **Обязательно: согласие на обработку персональных данных** (чекбокс + ссылка на
  политику). Без согласия заявка не отправляется. Юридический момент учесть.

## 7. Требования к данным (без проработки полной схемы БД)

> Полная схема БД проектируется отдельно. Здесь — обязательные требования.

- **Поле `status` у каждого продукта** (напр.: активен / черновик / скрыт / устарел).
- **Поле `status` у каждого банка** (напр.: активен / неактивен).
- **Поле «банк-партнёр»** у банка (опциональный атрибут, см. раздел 6).
- **Email банка** для доставки заявок.
- На MVP данные публикуются **без ручного ревью**, но статусы позволяют
  скрывать/модерировать записи при необходимости.
- Хранение заявок с привязкой к продукту и банку.

---

## 8. Вне рамок MVP

- Авторизация и личный кабинет пользователей.
- **Админка** — на MVP отсутствует (продукты только из парсера, управление через БД).
- **Ручное ревью** распарсенных данных — отсутствует (есть статусы для скрытия).
- Прочие категории: страхование, ипотека, карты, переводы (только кредиты + депозиты).
- Мобильное приложение.
- Платёжная интеграция.

## 12. Дополнения по ходу реализации (актуализация)

> Разделы ниже отражают решения, принятые при разработке и уже частично внедрённые. Дополняют исходное ТЗ, не отменяя его.

### 12.1. Фактический стек (уточнение к разделу 9)
- **Бэкенд:** Laravel **13** (требует PHP **8.4** — задан в `backend/Dockerfile`).
- **AI-провайдер:** через **OpenRouter** (OpenAI-совместимый, `response_format: json_schema strict`). Ключ в корневом `.env` как `OPENROUTER_API_KEY` или `AI_API_KEY` (docker-compose принимает оба). Рекомендуемая платная модель: `deepseek/deepseek-v3.2` (~$0.5 за полный обход). Free-tier (`*:free`) нестабилен (429, обрезанный JSON, мёртвые id) — годен только для дымовых тестов.
- **Скрейпер:** `jina` работает без ключа (keyless), `firecrawl` требует ключ.

### 12.2. Подкатегории продуктов
Каждый продукт имеет `subcategory` (определяется AI при парсинге, гибрид: AI-классификация + подсказка из раздела меню сайта).
- **credit:** `consumer | mortgage | auto | business | agro | education | refinance | pawn`
- **deposit:** `term | savings | demand | kids`
- общий fallback: `other`; для `installment` — `null`.
Хранение: `products.subcategory` (VARCHAR + CHECK). На витрине — бейдж и фильтр по подкатегории.

### 12.3. Категория `installment` (рассрочка / исламское финансирование)
Помимо `credit`/`deposit` введена третья категория **`installment`** — для банков, работающих по исламской схеме (наценка вместо процента): Alif, Tawhid. У таких продуктов ставка может быть 0 (беспроцентная рассрочка).

### 12.4. Рейтинги и отзывы банков
- Пользователь может **оставить отзыв о банке** (1–5★, текст, имя, обязательный consent).
- **Премодерация:** новый отзыв `status=pending`, попадает в выдачу и в рейтинг только после `approved` (вручную через БД на MVP).
- **Рейтинг банка** = AVG/COUNT по одобренным отзывам; показывается ★ в каталоге и на странице банка.
- Таблица `bank_reviews`. Эндпоинты: `GET/POST /api/banks/{id}/reviews`.

### 12.5. Курсы валют (по банкам)
- Виджет/раздел **курсов валют**: у каждого банка свои курсы покупки/продажи (TJS/USD/EUR и др.) — **парсятся** с сайтов банков (отдельный тип источника).
- Планируемое хранение: `bank_currency_rates` (bank_id, currency, buy, sell, rate_date). (Схема/парс — в работе, не внедрено.)

### 12.6. Email доставки заявок (уточнение к разделу 6)
Адрес доставки лида хранится **на источнике** `bank_source_urls.email` (по категории: у кредита/депозита может быть свой), а **не** на банке. `banks.contact_email` — публичный справочный контакт из реестра НБТ (НЕ для доставки).

### 12.7. Поля банка из реестра НБТ
В `banks` добавлены: `slug`, `website`, `phone`, `address_ru`, `address_tg`, `contact_email`. Источник списка и контактов — реестр НБТ (`nbt.tj/ru/banking_system/banks.php`). 18 банков засеяны.

### 12.8. Решения по схеме БД (зафиксированы)
- Тарифная сетка — **отдельная таблица `product_rates`** (Вариант A) + денормализованные `products.rate_min/rate_max`.
- **1 продукт = 1 валюта** (мультивалютный продукт банка дробится парсером на N записей).
- enum-поля — **VARCHAR + CHECK** (не нативный Postgres ENUM).
- `products.status` стартует как **`draft`**; публикуется после ручного `active`.
- `products.amount_min` — **nullable** (NULL = мин. сумма не указана; частый случай у кредитов).
- `products.term_min` — может быть `null` (тогда на витрине «до N мес.»).

### 12.9. Парсинг каталогов (index-режим)
Парсер сам определяет, является ли страница каталогом-меню (тогда обходит ссылки на детальные страницы продуктов) или прямой страницей продукта. Лимит 40 детальных страниц на источник, только домен банка.

### 12.10. Дизайн витрины
UX-паттерны агрегатора (референс — sravni.ru, **без копирования** их визуала/ассетов; своя палитра eskhata):
- **Каталог** — карточки-строки: банк + ★рейтинг, колонки Ставка/Сумма/Срок, чипы-фичи, CTA «Оставить заявку» + «В сравнение». Фильтры-сайдбар, мультиселект банков иконками с тоггл-кликом.
- **Главная** (планируется) — лендинг с плитками: Кредиты / Депозиты / Курсы валют / Оценка банков.
- Сравнение продуктов бок о бок.
